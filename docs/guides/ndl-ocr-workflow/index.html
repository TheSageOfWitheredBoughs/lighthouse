01

システム概要What this system does

本システムは、国立国会図書館デジタルコレクション（約400万点の資料を公開するデジタルアーカイブ）から、指定した文献のテキストを自動取得し、整形されたWord文書（.docx）とMarkdownファイルを生成して、あなたのPC・クラウドストレージに保存する一連のツール群です。

セットアップ手順Installation

セットアップは3ステップです。setup.ps1 が大部分を自動化するため、手作業は最小限で済みます。

scriptsフォルダをダウンロード

このガイドページと同じ場所に scripts/ フォルダが公開されています。GitHubリポジトリをクローンするか、ZIPでダウンロードして、PCの任意の場所に配置してください。

# 方法A: GitHubからクローン（推奨）
git clone https://github.com/[リポジトリURL].git
cd lighthouse/docs/guides/ndl-ocr-workflow/scripts

# 方法B: ZIPダウンロード後に展開
# ブラウザでGitHubの「Code → Download ZIP」からダウンロードPowerShell

setup.ps1 を実行

PowerShellを管理者権限で開き、scriptsフォルダに移動して setup.ps1 を実行します。このスクリプトは以下を自動的に行います：

  • Python と Git の存在確認
  • NDLOCR-Lite リポジトリのクローン（scripts/ndlocr-lite/ に配置）
  • 必要なPythonパッケージのインストール（python-docx, onnxruntime, opencv-python-headless 等）
# PowerShell の実行ポリシーを変更（初回のみ）
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned

# scriptsフォルダに移動
cd C:\path\to\scripts

# セットアップ実行
.\setup.ps1PowerShell
初回実行時の所要時間

NDLOCR-Liteのクローンにはモデルファイル（ONNX形式、合計約150MB）が含まれるため、ネットワーク環境によっては数分〜10分程度かかる場合があります。途中で中断しないでください。

config.py を編集

テキストエディタで config.py を開き、自分の環境に合わせて保存先を設定します。詳細は次のセクション（Section 04）で解説します。

Obsidianを使用している場合、OCR結果をMarkdownファイルとしてObsidian保管庫内に直接保存できます。YAML Frontmatter（タイトル・PID・取得ルート・日付・タグ）が自動付与されるため、Obsidianの検索・タグ・リンク機能でそのまま活用できます。

# config.py 内の該当箇所
OBSIDIAN_NDL_PATH = r"C:\Users\YourName\Documents\ObsidianVault\NDL_Archives"

指定したパスにフォルダが存在しない場合は自動作成されます。

Google Drive に保存する場合

Google Driveデスクトップ版がインストールされている場合に有効化します。内部では gws コマンド（Google Workspace CLI）を使用してアップロードを行います。

GDRIVE_ENABLED = True
GDRIVE_BASE_FOLDER = "NDL_Archives"  # マイドライブ直下に作成されるフォルダ名
gws コマンドについて

Google Drive API を使うために gws コマンドが必要です。pip install gws でインストールできます。gws がない場合でも、他の保存先（Obsidian、Dropbox、ローカル）は正常に動作します。

Dropbox に保存する場合

Dropboxデスクトップアプリがインストールされている場合、同期フォルダに直接ファイルを書き込むことで保存します。APIやCLIツールは不要で、ファイルシステム上のパスを指定するだけです。

DROPBOX_ENABLED = True
DROPBOX_PATH = r"C:\Users\YourName\Dropbox\NDL_Archives"

大学のDropboxアカウントでデータ管理をしている場合は、ここを有効にすると大学の環境からもアクセスできるようになります。

AI_REPAIR_MAP = {
    "國": "国", "眞": "真", "實": "実", "來": "来",
    "敎": "教", "辨": "弁", "吿": "告", "淨": "浄",
    "佛": "仏", "如來": "如来", "信樂": "信楽",
    # 独自のパターンを自由に追加可能
    # "誤字パターン": "正しい文字",
}
補正辞書のカスタマイズ

自分が頻繁に扱う分野の文献でOCR誤読が繰り返し発生するパターンがあれば、ここに追加することで以後のすべてのOCR処理に自動適用されます。辞書を空にしたい場合は AI_REPAIR_MAP = {} と書いてください。

05

アーキテクチャ解説How it works

本システムの処理フローは、NDLにOCRテキストが既に存在するかどうかによって2つのルートに自動分岐します。ユーザーはどちらのルートが選択されるかを事前に知る必要はありません。スクリプトが自動判定します。

全体処理フロー

Step 1: PID解析 + 書誌情報取得
入力されたURL/PIDからNDLサーチAPIで書誌情報（タイトル・著者）を取得します。Book APIも併用して、メタデータの信頼性を高めます。
Step 2: OCRテキスト存在確認（分岐点）
NDL次世代デジタルライブラリーのBook API + Page APIを順にチェックし、既存のOCRテキストが利用可能かを判定します。
APIルート（高速）
既にNDL側でOCR処理済みのテキストが存在する場合。全文API（gzip圧縮）で一括取得を試み、失敗時はページ単位のAPI取得にフォールバックします。著作権保護期間満了の資料（約28万点）の多くがこちらに該当します。
OCRルート（ローカル処理）
NDLにテキストがない場合。IIIF API経由で画像をダウンロードし、NDLOCR-Lite（ローカルのONNXモデル）でリアルタイムOCR処理を行います。1ページあたり30〜60秒かかります。
Step 3: テキスト補正
config.py の AI_REPAIR_MAP に基づく旧字体補正が自動適用されます。Antigravityと連携している場合は、この段階でAIによる高度な文脈校正（助詞の補完、句読点の復元等）も行われます。
Step 4: ファイル保存
config.py で有効化されたすべての保存先（Obsidian / Dropbox / Google Drive / ローカル）に、Markdownファイル（YAML Frontmatter付き）を書き出します。
Step 5: 結果レポート
処理結果（使用ルート・取得文字数・保存先パス）をコンソールに出力します。[RESULT_JSON] 以降にJSON形式のデータも出力されるため、Antigravityが後続処理（DOCX生成等）に利用できます。

NDL API の構造

本システムは、国立国会図書館が提供する複数のAPIを組み合わせて使用しています。以下はそれぞれのAPIの役割と、システム内での使用箇所の対応表です。

API名エンドポイント役割使用箇所
NDLサーチ RSS ndlsearch.ndl.go.jp/api/opensearch 書誌情報（タイトル等）の取得 Step 1
Book API lab.ndl.go.jp/dl/api/book/{pid} 資料メタデータ・ページ一覧の取得 Step 1, 2
Fulltext API lab.ndl.go.jp/dl/api/book/fulltext/{pid} 全文テキスト一括取得（gzip/zip） Step 2a（APIルート）
Page API lab.ndl.go.jp/dl/api/page/{pid}_{page} ページ単位のOCRテキスト取得 Step 2a（フォールバック）
IIIF Image API dl.ndl.go.jp/api/iiif/{pid}_{seq}/full/full/0/default.jpg ページ画像のダウンロード Step 2b（OCRルート）

OCRエンジンの仕組みNDLOCR-Lite internals

NDLOCR-Liteは、国立国会図書館（NDLラボ）が2024年に公開した軽量OCRエンジンです。GPUを必要とせず、一般的なPCのCPUのみで動作するよう設計されています。内部では2段階のONNXモデルによるカスケード処理が行われます。

OCR処理パイプライン

Stage 1: レイアウト認識（DEIMv2）
入力画像（JPG/PNG/TIFF/BMP）に対して、深層学習モデル DEIMv2（deim-s-1024x1024.onnx）がテキスト行の矩形領域を検出します。同時に、各テキスト行の文字数カテゴリ（少/中/多）を判定し、次段のPARSeqモデルの選択に使用します。縦書き・横書きの混在レイアウトにも対応しています。
Stage 2: 文字認識（PARSeqカスケード）
検出された各テキスト行に対して、文字数カテゴリに応じた最適なPARSeqモデルが選択・適用されます。3つのモデルがカスケード構成で動作します：
parseq-ndl-30
文字数が少ない行（30文字以下）向け。高速処理に最適化されており、短い見出しやキャプションに使用されます。
parseq-ndl-50
中程度の行（31〜50文字）向け。一般的な本文行の多くがここに該当します。
parseq-ndl-100
長い行（51〜100文字）向け。密度の高い活字や注釈部分に使用されます。認識精度が最も高いモデルです。
Stage 3: 読み順整序
認識されたテキスト行を、原本のレイアウトに基づいて正しい読み順に並べ替えます。日本語の古典籍に特徴的な縦書き右→左の読み順に対応しており、段組みや割注（小さい文字の注釈）も適切に処理されます。

処理速度の目安

処理内容1ページあたり100ページあたり
画像ダウンロード2〜5秒3〜8分
OCR処理（CPU）30〜60秒50分〜1.5時間
テキスト出力即時即時
APIルートとの使い分け

NDLが既にOCR処理済みのテキストを公開している資料（著作権保護期間満了の約28万点）では、OCRを自分で実行する必要はありません。本システムは自動的にAPIルートを選択し、数秒でテキストを取得します。OCRルートが起動するのは、APIにテキストがない資料（著作権保護中のためOCR非公開の資料等）のみです。

08

DOCX生成の仕組みDocument generation

ndl_docx_generator.py は、OCR/APIで取得したプレーンテキストを、読みやすく構造化されたWord文書に変換するスクリプトです。内部で以下の処理を行います。

自動見出し検出

テキスト中の「第一章」「自序」「目次」「解題」等の見出しパターンを正規表現で自動検出し、Word文書の見出しスタイル（Heading 1）として整形します。見出しの直前には自動改ページが挿入されるため、章ごとに新しいページから始まります。

# 自動検出される見出しパターン
第一章 / 第二編 / 第三回 / 第四節
自序 / 目次 / 要目索引 / 解題 / 序文 / 後記 / 付録

デザイン仕様

要素設定カスタマイズ
本文フォントMS 明朝config.py の DOCX_BODY_FONT
本文サイズ18ptDOCX_BODY_SIZE または --body-size
見出しフォントMS ゴシックDOCX_HEADING_FONT
見出しサイズ22ptDOCX_HEADING_SIZE または --heading-size
見出し色濃紺 (#000080)DOCX_HEADING_COLOR
行間1.5倍DOCX_LINE_SPACING または --line-spacing
段落配置両端揃え固定
字下げ本文サイズ分（1文字分）自動
ページ参照各ページ右上に [NDL p.XX]自動

段落統合処理

OCRは原本の物理的な改行位置でテキストを分割するため、論理的には1つの段落が複数行に分割されることがあります。ndl_docx_generator.py はこれを検知し、連続するテキスト行を1つの論理段落に統合します。空行は段落の区切りとして認識されます。

Antigravity連携Workflow integration

Antigravity（Gemini CLI）がインストールされている場合、/ndl コマンドとして本システムを直接呼び出せるようになります。

導入手順

ワークフローファイルをコピー

scripts/workflows/ndl.md を、Antigravityの .agents/workflows/ フォルダにコピーします。

Copy-Item .\workflows\ndl.md ~\.agents\workflows\ndl.mdPowerShell

Antigravityで実行

Antigravityのチャットで /ndl コマンドを入力するだけで、テキスト取得からAI校正、DOCX生成、保存までが一連の対話フローで実行されます。

# Antigravityチャット内で入力

AntigravityによるAI校正

Antigravityと組み合わせた場合の最大の利点は、AI校正ステップ（Step 2）が追加されることです。ndl_vault_sync.py のStep 1で取得した生テキストに対し、AntigravityのAIが以下の校正を行います：

  • OCR誤認識の文脈的修正（前後の文脈から正しい文字を推定）
  • 旧字体の可読化（意味・ニュアンスが変わるものは原文保持）
  • 助詞・句読点の補完（OCRによる欠落箇所を復元）
  • 段落構造の整形（OCR由来の不要な改行を統合）
Antigravityなしでも動作します

Antigravityがインストールされていなくても、ndl_vault_sync.py と ndl_docx_generator.py はスタンドアロンのPythonスクリプトとして動作します。AI校正ステップがスキップされ、config.py の辞書ベース補正のみが適用される形になります。

トラブルシューティングTroubleshooting

エラーが発生した場合は、以下のパターンを確認してください。ほとんどの問題は環境設定またはネットワークに起因します。

「config.py が見つかりません」と表示される

原因: ndl_vault_sync.py と config.py が同じフォルダにありません。

解決策: 両方のファイルが scripts/ フォルダ内に存在することを確認してください。カレントディレクトリが scripts/ であることも確認してください。

「python: command not found」「python は認識されていません」

原因: PythonにPATHが通っていません。

解決策A: Pythonインストーラーを再実行し、「Add Python to PATH」にチェックを入れて「Modify」を選択します。

解決策B: config.py の PYTHON_BIN にフルパスを指定します。

PYTHON_BIN = r"C:\Users\YourName\AppData\Local\Programs\Python\Python312\python.exe"
「ModuleNotFoundError: No module named 'docx'」

原因: python-docx パッケージがインストールされていません。

解決策: pip install python-docx を実行してください。setup.ps1 を実行済みであればこのエラーは発生しないはずですが、Pythonの仮想環境を使用している場合は、正しい環境がアクティブになっているか確認してください。

「API取得失敗」「テキスト取得に失敗しました」

原因A: ネットワーク接続の問題。NDLのサーバーに到達できていません。

原因B: 指定したPIDの資料が存在しないか、アクセス制限があります。

解決策:

  • ブラウザで https://dl.ndl.go.jp/pid/[PID] にアクセスし、資料が存在するか確認してください。
  • NDLのサーバーが一時的にダウンしている可能性もあります。数分後に再試行してください。
  • VPNやプロキシ環境下では接続がブロックされる場合があります。
OCRルートで「NDLOCR-Lite エラー」が発生する

原因A: ONNXモデルファイルが不完全（ダウンロードが中断された等）。

原因B: onnxruntime のバージョン不整合。

解決策:

  1. scripts/ndlocr-lite/ フォルダを削除し、setup.ps1 を再実行してクリーンクローンを行ってください。
  2. pip install --upgrade onnxruntime で最新版にアップデートしてください。
保存先にファイルが作成されない

原因: config.py の保存先設定が正しくありません。

確認事項:

  • OBSIDIAN_NDL_PATH のパスが実在するか（存在しない場合は自動作成されますが、親ディレクトリは存在する必要があります）
  • Dropboxの場合、同期クライアントが起動しているか
  • Google Driveの場合、gws コマンドがインストールされ、認証済みか
  • すべての保存先が無効の場合は、Documents/NDL_Archives/ にフォールバック保存されます
