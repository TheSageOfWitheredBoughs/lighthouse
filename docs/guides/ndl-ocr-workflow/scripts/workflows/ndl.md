# /ndl ワークフロー

NDLデジタルコレクションのURL/PID指定、またはローカルPDFファイルパス指定で、
文字起こしを全自動で保存先（Obsidian / Dropbox / Google Drive / ローカル）に保存します。

**対応ルート:**
- **NDL API ルート**: NDLが持つOCRテキストをAPIから直接取得（高速）
- **NDL OCR ルート**: NDL IIIF画像をダウンロード → NDLOCR-Lite でOCR
- **PDF テキスト抽出ルート**: PDFのテキスト層を直接抽出（高速・高精度）
- **PDF OCR ルート**: スキャンPDFをページ画像化 → NDLOCR-Lite でOCR

## 使い方

```
# --- NDLデジタルコレクション ---
/ndl https://dl.ndl.go.jp/pid/3048008
/ndl --pid 3048008 --pages 1-10
/ndl --pid 3048008 --pages 1 --no-gdrive

# --- ローカル / ダウンロードPDF ---
/ndl --pdf /path/to/downloaded_book.pdf
/ndl --pdf /path/to/book.pdf --pages 1-30
/ndl --pdf /path/to/book.pdf --title "明治民法草案" --pages 1-50
```

// turbo-all

### Step 1: スクリプト実行（テキスト取得・判定）

**NDLデジタルコレクションの場合:**
```powershell
python ndl_vault_sync.py --pid {PID} --pages {PAGES}
```

**ローカル / ダウンロードPDFの場合:**
```powershell
python ndl_vault_sync.py --pdf "{PDFファイルの絶対パス}" --pages {PAGES}
python ndl_vault_sync.py --pdf "{PDFファイルの絶対パス}" --title "{書名}" --pages {PAGES}
```

※ config.py の設定に基づいてパスが自動解決されます。

実行後、以下の情報がコンソールに出力されます：
- **ルート判定結果**（API / OCR / PDF_TEXT / PDF_OCR）
- **先頭500文字のプレビューテキスト**
- **`[RESULT_JSON]`** 以降にJSON形式の全データ

### Step 1.5: PDFルートでの書名確認（`--pdf` 使用時のみ）

`--title` を省略した場合、書名はPDFファイル名から自動設定されます。
必要に応じてユーザーに書名を確認し、`--title "正式な書名"` を付けて再実行します。

### Step 2: AI校正・補完（私が実施）

Step 1の出力の `raw_text` を読み取り、以下の校正を実施します：
- OCR誤認識の修正（文脈から判断）
- 旧字体の可読化（意味に差異のない一般的な置換のみ実施）
- 文脈上の欠落補完（助詞・句読点・専門用語の復元）
- 段落構造の整形

校正完了後、保存先のMarkdownファイルを更新します。

### Step 3: 高品質DOCX変換（標準実施）

```powershell
python ndl_docx_generator.py --input [テキストファイル] --output [出力先.docx] --title "[タイトル]" --pid "[PID]"
```

### Step 4: OCRルート時のGoogle Drive確認（該当する場合のみ）

ルートが「OCR」だった場合のみ、サブフォルダ作成の確認を求めます。

### Step 5: 最終確認レポート出力

以下を日本語でレポートします：
- 使用ルート（API取得 / NDLOCR-Lite OCR）
- 書誌情報（タイトル、PID、ページ範囲）
- 取得文字数
- 保存先パス
- DOCX仕様の適用確認

## ⛔ 制動規則

### 自動停止条件
- **スクリプトエラー**: 例外終了した場合、再試行せず補正プロトコルへ移行する
- **OCR品質**: OCR出力テキストが空白・文字化けのみの場合は停止し、画像品質の問題を報告する
- **ファイルサイズ異常**: 生成されたDOCXが1KB未満の場合はファイル破損とみなし報告する
- **連続エラー**: 同一ステップで2回連続してエラーが発生した場合は停止して診断結果を報告する

### Deny List（絶対に実行しないコマンド・操作）
- `rm`・`del`・`Remove-Item` 系コマンドによるファイル削除
- `--force` フラグを伴う git コマンド
- 著作権保護期間中資料の画像ファイルをローカル外へ転送・共有すること

## 🔧 補正プロトコル（エラー発生時）

エラー発生時は**即座に再試行せず**以下の順序で対応する。

### Step A: 診断
- どのステップで何のエラーが発生したかを記録する
- スクリプトのエラーメッセージ全文をチャットに出力する

### Step B: 仮説立案（最低2案を立てる）

| 状況 | 仮説1（可能性高） | 仮説2（代替） |
|------|----------------|-------------|
| スクリプトエラー | Python環境またはライブラリの不整合 | NDL API の仕様変更・レート制限 |
| OCRルートで空白出力 | 入力画像の解像度が低すぎる | モデルファイルが破損 |
| DOCX生成エラー | python-docx ライブラリの依存関係 | 出力先パスに書き込み権限なし |
| 保存エラー | クラウドストレージ同期の競合ロック | 保存先フォルダが存在しない |
| PDF_TEXT が空白 | PDFが画像のみのスキャン（テキスト層なし） | PDFが暗号化・パスワード保護されている |
| PDF_OCR が空白 | スキャン画像の解像度が低い（300DPI未満） | NDLOCRモデルが日本語以外の文字に対応不可 |
| pypdfium2 ImportError | `pip install pypdfium2` 未実行 | Python環境が32bit（pypdfium2は64bit必須） |

### Step C: 検証（最大2回まで試行）
1. 可能性の高い仮説から診断コマンドを実行して確認する
2. 修正後に再実行する
3. 2回試行後も解決しない場合は**停止**し報告する

### Step D: 証跡記録
補正プロトコルを実行した場合、WOR Loop の「ボトルネックと気づき」に記録する。

## 注意事項

- 著作権保護期間中の資料はAPIルートが利用できず、OCRルートになります
- OCRの処理時間はページ数とPC性能によって変わります（1ページあたり目安：30〜60秒）
- PIDはNDLデジタルコレクションのURLから自動抽出されます（手入力も可）
- DOCX変換は全資料に標準適用します
